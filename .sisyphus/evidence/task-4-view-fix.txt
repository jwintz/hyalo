Task 4: Fix ios_report_bootstrap_complete view timing issue
============================================================

Problem
-------
Emacs boots successfully on iOS, but the view is nil when ios_report_bootstrap_complete
is called from the Emacs thread. The error message was:

  "ios_report_bootstrap_complete: ERROR - view is nil, cannot schedule redraw!"

Root Cause
----------
The bootstrap completion callback fires BEFORE the EmacsView is created by the
frame initialization code. This is a race condition between:

1. Emacs thread: Running ios_emacs_init() → bootstrapping Lisp → calling ios_report_bootstrap_complete()
2. Main thread: Creating frame → Creating EmacsView → calling ios_set_main_emacs_view()

The timing was such that bootstrap completion happened before the view existed.

Solution
--------
Modified iosterm.m to handle the timing mismatch gracefully:

1. Added a pending redraw flag (ios_bootstrap_complete_pending_redraw)
2. Modified ios_report_bootstrap_complete():
   - If view exists: schedule redraw immediately
   - If view is nil: set pending flag instead of erroring
3. Modified ios_set_main_emacs_view():
   - After setting the view, check if a redraw is pending
   - If pending: trigger the redraw immediately

This ensures the redraw happens regardless of which comes first.

Files Modified
--------------
- ~/Syntropment/hyalo-feedstock-unified/emacs/src/iosterm.m
- ~/Syntropment/hyalo-feedstock-unified/ios/iosterm.m

Key Changes
-----------

1. Added pending flag:
   static bool ios_bootstrap_complete_pending_redraw = false;

2. ios_report_bootstrap_complete() now:
   - Sets ios_bootstrap_complete_pending_redraw = true when view is nil
   - No longer logs an error - this is an expected condition

3. ios_set_main_emacs_view() now:
   - Checks ios_bootstrap_complete_pending_redraw after setting view
   - Triggers redraw if bootstrap already completed

After the fix:

- The redraw is deferred until the view is available
